## 实施策略

- 策略：先做首版验证版。
- 选择理由：需求是从零搭建本地盯盘工具，首要目标是跑通“行情/回放 -> 1 分钟/5 分钟 -> 指标 -> 规则候选 -> 大模型复核 -> Web 展示”的闭环；自动下单、全市场扫描和复杂成本核算都不在首版范围。
- 切换条件：当 1-5 只股票的回放和模拟实时链路稳定后，再扩展专业行情源、通知渠道或更复杂的交易记录。
- 生成约束：前端用 `pnpm create vite` 脚手架生成 React + TypeScript 工程；后端用 Python/FastAPI 标准工程结构与依赖管理搭建，不手写前端工程骨架。

## 技术栈

- 后端：Python 3.12、FastAPI、Pydantic、Pandas、AKShare、OpenAI-compatible HTTP client、pytest。
- 前端：Vite、React、TypeScript、Tailwind CSS、Recharts/lightweight chart library、Vitest。
- 通信：REST 配置接口 + Server-Sent Events 或 WebSocket 实时事件。
- 存储：首版使用本地 JSON/SQLite 二选一，优先 SQLite；行情窗口主要保存在内存并支持回放测试。

## 交付切片

- Slice 1：脚手架与基础运行闭环。生成前端工程，建立后端工程结构，提供健康检查和前端页面。
- Slice 2：行情与 K 线核心。实现标准 K 线模型、1 分钟缓存、5 分钟聚合、历史/模拟回放。
- Slice 3：指标与规则信号。实现 VWAP、量能、相对强弱、三低、急跌量缩、急拉不封板、VWAP 高抛低吸规则。
- Slice 4：大模型复核与风控降级。候选点触发模型，结构化输出校验，失败降级，数据延迟暂停信号。
- Slice 5：Web 盯盘台。股票池、图表、持仓输入、信号卡、大模型理由、人工操作记录。

## 开发顺序

1. 使用脚手架生成 `frontend`。
2. 初始化 `backend` 的依赖文件和标准目录。
3. 按 TDD 做后端核心模型、聚合、指标、规则、LLM 适配。
4. 接 FastAPI 与事件流。
5. 构建前端仪表盘并接入模拟/回放数据。
6. 使用本地浏览器检查 UI 和交互。

## 测试与验证

- 后端：
  - `python -m pytest`
  - 覆盖模型、聚合、指标、规则、LLM 降级、API。
- 前端：
  - `pnpm build`
  - `pnpm test` 若 Vitest 已配置。
- 联调：
  - 后端回放/模拟流启动。
  - 前端能显示 1-5 只股票、持仓、候选信号、大模型复核状态。
  - 数据延迟时前端显示 `data_delayed` 并暂停新信号。

## 风险清单

| 风险 | 影响 | 缓解 | 负责人 |
|------|------|------|--------|
| 免费行情源延迟或不可用 | 盘中无法稳定出信号 | 保留回放/模拟模式，Provider 健康状态可见 | 开发 |
| 策略阈值不适配不同股票 | 买卖点过多或过少 | 阈值配置化，信号必须带原因和风险 | 开发/使用者 |
| 大模型输出不可解析 | 信号卡阻塞 | 强制 JSON schema，失败降级为规则候选 | 开发 |
| 前端图表复杂度过高 | 首版拖慢交付 | 首版先用稳定图表库和简化指标卡 | 开发 |
| 用户误把提示当自动建议 | 决策风险 | UI 明确“只提示、不自动交易、需人工确认” | 开发 |
